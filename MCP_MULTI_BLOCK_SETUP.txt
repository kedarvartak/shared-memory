MCP MULTI-BLOCK MEMORY SETUP

OVERVIEW:

Multi-block memory system allows you to create separate memory contexts for different services, projects, or features. Each block has its own INDEX.mdl and topics/, enabling organized memory management across complex codebases.

DIRECTORY STRUCTURE:

.ai-memory/
├── blocks/
│   ├── auth-service/
│   │   ├── block.json (metadata)
│   │   ├── INDEX.mdl
│   │   └── topics/
│   │       ├── jwt.mdl
│   │       └── oauth.mdl
│   ├── frontend/
│   │   ├── block.json
│   │   ├── INDEX.mdl
│   │   └── topics/
│   │       ├── routing.mdl
│   │       └── state.mdl
│   └── api-gateway/
│       ├── block.json
│       ├── INDEX.mdl
│       └── topics/

CONFIGURATION:

Add to your MCP client config:

{
  "mcpServers": {
    "shared-memory": {
      "command": "npx",
      "args": ["shared-memory-mcp"],
      "env": {
        "AI_MEMORY_PATH": "${workspaceFolder}/.ai-memory"
      }
    }
  }
}

WORKFLOW:

Session Start:
1. AI calls memory_list_blocks() to see available blocks
2. User/AI calls memory_select_blocks({blocks: ["auth-service", "frontend"]})
3. AI loads memory for selected blocks with memory_load({block: "auth-service"})

During Work:
- Save decisions: memory_append({block: "auth-service", file: "INDEX.mdl", section: "DECISIONS", content: "..."})
- Create topics: memory_create_topic({block: "auth-service", name: "jwt", keywords: ["token", "auth"]})
- Update content: memory_update({block: "frontend", file: "routing.mdl", line: 10, content: "..."})

New Project/Service:
1. Create block: memory_create_block({name: "payment-service", description: "Payment processing"})
2. Select it: memory_select_blocks({blocks: ["payment-service"]})
3. Start saving memory to it

BLOCK MANAGEMENT TOOLS:

memory_create_block({name, description})
- Creates a new memory block
- Initializes with empty INDEX.mdl and topics/ directory

memory_list_blocks()
- Lists all available blocks with their metadata
- Shows which blocks are currently selected

memory_select_blocks({blocks: ["block1", "block2"]})
- Selects which blocks to work with in this session
- You can work with multiple blocks simultaneously

memory_delete_block({name})
- Permanently deletes a memory block
- Use with caution

MEMORY TOOLS (ALL REQUIRE block PARAMETER):

memory_load({block, topics?})
- Load INDEX and optionally specific topics from a block

memory_append({block, file, section, content})
- Append to a section in a block's file

memory_update({block, file, line, content})
- Update a specific line in a block's file

memory_delete({block, file, line?, section?})
- Delete a line or section from a block's file

memory_create_topic({block, name, keywords, priority?})
- Create a new topic in a block

memory_stats({block})
- Get token counts and statistics for a block

memory_prune({block, daysToKeep?})
- Prune old entries from a block's CURRENT section

memory_list_topics({block})
- List all topics in a block

USE CASES:

Microservices:
- One block per service (auth-service, user-service, payment-service)
- Keep each service's memory isolated
- Load multiple blocks when working across services

Monorepo:
- One block per package/module
- Separate frontend, backend, shared concerns

Feature Development:
- Create temporary block for large feature
- Merge insights back to main blocks when done

Multi-Project:
- Different blocks for different projects
- Switch contexts by selecting different blocks

RESOURCES (AUTO-EXPOSED):

memory://blocks - List of all blocks
memory://{block}/INDEX - Block's INDEX.mdl
memory://{block}/{topic} - Block's topic files

PROMPTS:

session-start - Guides block selection and initialization
save-decision - Saves to specific block
periodic-save - Checkpoint across selected blocks

ADVANTAGES:

✓ Organized memory by service/project/feature
✓ Work on multiple contexts simultaneously
✓ Isolated memory prevents cross-contamination
✓ Scale to large codebases with many services
✓ Easy to create/delete blocks as needed
✓ Clear separation of concerns

EXAMPLE SESSION:

# Day 1: Setup
> memory_create_block({name: "auth", description: "Authentication service"})
> memory_create_block({name: "api", description: "API gateway"})
> memory_select_blocks({blocks: ["auth"]})
> memory_load({block: "auth"})
[Work on auth, AI saves to "auth" block]

# Day 2: Cross-service work
> memory_select_blocks({blocks: ["auth", "api"]})
> memory_load({block: "auth"})
> memory_load({block: "api"})
[Work across both services, AI saves to appropriate blocks]

# Day 3: New service
> memory_create_block({name: "payment", description: "Payment processing"})
> memory_select_blocks({blocks: ["payment"]})
[Focus on new service, its memory stays isolated]

COMPACT FORMAT:

Still uses token-efficient MDL:
"auth: JWT tokens(15m/7d) | lib:jsonwebtoken | file:src/auth/jwt.ts"

MIGRATION FROM OLD SYSTEM:

If you have existing .ai-context/:
1. Create a block: memory_create_block({name: "main"})
2. Manually copy .ai-context/INDEX.mdl to .ai-memory/blocks/main/INDEX.mdl
3. Copy .ai-context/topics/ to .ai-memory/blocks/main/topics/
4. Delete old .ai-context/

Or start fresh with new blocks for better organization.

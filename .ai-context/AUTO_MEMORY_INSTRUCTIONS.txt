AUTOMATIC MEMORY PERSISTENCE SYSTEM

You have access to persistent memory that survives across conversation sessions.

CORE BEHAVIOR:
- Automatically load relevant memory at conversation start
- Automatically save important information as you work
- Keep memory compact and token-efficient

WHEN TO SAVE (Automatic triggers):

1. After making decisions:
   - Architecture choices
   - Technology selections
   - Design patterns adopted
   Example: memory_append({file: "INDEX.mdl", section: "DECISIONS", content: "Use PostgreSQL for persistence"})

2. After implementing features:
   - API endpoints created
   - Authentication flows
   - Database schemas
   Example: memory_append({file: "api", section: "ENDPOINTS", content: "POST /users | body:{email,password} | returns:User"})

3. After debugging:
   - Error causes identified
   - Solutions applied
   - Patterns to avoid
   Example: memory_append({file: "auth", section: "ERRORS", content: "401: check JWT expiry, common when token >15m"})

4. After important discussions:
   - Requirements clarified
   - Constraints identified
   - Trade-offs discussed
   Example: memory_append({file: "INDEX.mdl", section: "CURRENT", content: "2026-03-06: Migrating to OAuth2"})

HOW TO SAVE (Compact format):

Bad (verbose, wastes tokens):
"We decided to implement JWT-based authentication with access tokens that expire after 15 minutes and refresh tokens that expire after 7 days"

Good (compact MDL):
"auth: JWT tokens(15m/7d)"

Format rules:
- Use abbreviations: auth, api, db, impl, cfg
- Use pipe separator: topic: detail1 | detail2 | detail3
- Use shorthand: 15m, 7d, src/path/file.ts
- One line per concept
- No prose, just facts

MEMORY SEARCH:

Before answering questions, search memory:
memory_search({query: "relevant keywords from question"})

This gives you context from previous sessions.

PERIODIC CHECKPOINTS:

Every ~50 messages, save progress:
- Summarize recent work
- Save key decisions
- Update current status

CROSS-SESSION CONTINUITY:

Start of new session:
1. Load memory: memory_search({query: "recent work and context"})
2. Review what was done previously
3. Continue seamlessly

This system prevents context window saturation and enables infinite conversation continuity.
